# ESP32 Logger — Full Code Audit (20 Apr 2026)

## Scope and exclusions
- Reviewed firmware backend, runtime architecture, API surface, and SPA UI/UX.
- Explicitly **excluded** prior audit files and **excluded** `full_logger.ino` per request.
- Took into account your target board choice: **ESP32-C3 + custom balanced partition table**.

## Executive summary
The project is technically ambitious and above-average for an embedded+web stack: modular sensor architecture, task pipeline, fallback UX, and defensive controls already exist. The biggest current risks are:
1. **Security hardening gaps still open by default** (auth disabled by default, destructive actions via same-origin UI, large inline failsafe script).
2. **Flash/maintainability pressure** on ESP32-C3 because almost all sensors/exporters are compile-enabled by default.
3. **UI consistency debt** (mixed interaction patterns, duplicate DOM IDs, heavy `innerHTML` rendering surface).
4. **Config/runtime reliability edges** around platform config lifecycle and save/restart flows.

Overall rating (current state):
- Architecture: **8/10**
- Reliability: **7/10**
- Security: **6/10**
- UX consistency: **6.5/10**
- Production readiness for unattended deployments: **7/10** (after hardening can be 8.5+)

## 1) Backend / firmware architecture audit

### What is strong
- Clear pipeline decomposition (`SensorTask -> ProcessingTask -> Storage/Export`) with queue-based decoupling and priorities tuned for sensor timing.
- Runtime protections exist: queue depth tuning, watchdog-like health check, FS mutex, config mutex, wire mutex.
- Good storage strategy: hybrid SD/LittleFS with cloud-only option and fallback behavior.
- Thoughtful API data path with ring-buffer + FS merge + aggregation path.

### Key findings

#### [HIGH] Feature bloat vs ESP32-C3 flash/RAM budget
- `setup.h` enables nearly all sensors/exporters by default, increasing binary size and static init footprint.
- On your partition table (`app0/app1 = 0x180000` each, ~1.5MB per OTA slot), this is the main long-term risk for future updates.
- Recommendation:
  - Create profile-based builds (`minimal`, `env-basic`, `full-lab`) via `-D` flags.
  - Keep default release profile minimal and opt-in extra plugins.

#### [MEDIUM] Save/reload UX and backend coupling
- `/save_platform` returns `{"ok":true}` before full lifecycle result reaches UI; reload is triggered separately via `/api/platform_reload`.
- Works, but operationally fragile under unstable Wi‑Fi (user may read success while restart/reload path fails).
- Recommendation:
  - Add transactional result object (`saved`, `validated`, `reload_scheduled`) and display each state in UI.

#### [MEDIUM] Startup/config parse duplication
- Similar config file parse logic appears in multiple places (mode detect, sleep config, task manager storage config), increasing drift risk.
- Recommendation:
  - Centralize platform config parsing/validation in one manager with a typed in-memory snapshot.

## 2) Security audit (backend + web)

### What is strong
- CSP, `X-Frame-Options`, `nosniff`, `Referrer-Policy` headers are set.
- Optional Basic Auth gate exists and is globally applied when enabled.
- Global token-bucket limiter applied to mutating routes.
- Crash-safe temp-write + rename for `/save_platform`.

### Key gaps

#### [HIGH] Auth is compile-time optional and OFF by default
- `WEB_BASIC_AUTH_ENABLED` defaults to `0`.
- For AP-only lab use this is acceptable, but for any bridged/networked deployment this is high risk.
- Recommendation:
  - Enable auth by default for non-dev builds.
  - Add per-device bootstrap password and forced first-change flow.

#### [MEDIUM] Heavy inline JS/HTML in failsafe mode
- Failsafe page embeds very large inline script and uses prompt/confirm patterns.
- This increases auditing difficulty and XSS blast radius if any reflected/stored vector appears in failsafe flows.
- Recommendation:
  - Move failsafe JS/CSS to static assets with nonce/hash CSP for setup mode.

#### [MEDIUM] Broad `innerHTML` usage in SPA
- Main pages and sensors/settings views build UI via string concatenation + `innerHTML`.
- There is escaping in several places, but consistency is difficult to guarantee across all callsites.
- Recommendation:
  - Adopt small templating helper using DOM APIs (`createElement`, `textContent`) for user-originated fields.

## 3) API and data path audit

### What is strong
- `/api/data` has guardrails (limit clamp), streaming response, and merge-dedup strategy.
- Diagnostics endpoint exists (`/api/diag`) which is very useful in production.
- Ring-buffer + historical aggregation split is sensible for constrained RAM.

### Risks / improvements
- Add request correlation IDs in logs for mutating endpoints.
- Add explicit schema validation for `platform_config.json` (required fields, numeric bounds).
- Add checksum/version field to config writes and return it in `/api/platform_config` for optimistic UI concurrency control.

## 4) UI / UX audit (consistency, usability, accessibility)

### Strengths
- Responsive layout with sidebar + bottom nav strategy.
- Theme system is mature and starts early to avoid theme flash.
- Core flows (dashboard/live/settings/update/sensors) are easy to discover.
- Failsafe recovery UX is excellent for embedded reality.

### Issues

#### [MEDIUM] Duplicate DOM id in dashboard
- `id="db-error"` appears twice; this causes ambiguous selector behavior and inconsistent error rendering.
- Fix immediately.

#### [MEDIUM] Interaction model inconsistency
- App mixes delegated data-* handlers (good direction) with inline handler style in embedded failsafe HTML.
- Standardize on one event model.

#### [LOW-MEDIUM] Accessibility modernization needed
- Good base exists, but for WCAG 2.2 alignment:
  - Add stronger focus-visible states across controls.
  - Ensure every status message has ARIA live region strategy.
  - Validate color contrast in both light/dark custom themes.
  - Add keyboard test matrix for all settings panels.

## 5) ESP32-C3 + Custom Balanced Partition Table assessment

Your partition table:
- `app0/app1`: `0x180000` each (~1.5MB)
- `spiffs`: `0xF0000` (~960KB)

This is a sensible OTA/filesystem balance for this product class. Main practical implication:
- **Firmware growth margin is now your limiter**, not filesystem.

Recommendations tailored to this table:
1. Keep gzip-compressed web assets and strip unused locales/resources.
2. Build-time feature profiles (critical for OTA longevity).
3. Add CI size budget checks (fail build if firmware exceeds threshold, e.g. 85% of OTA slot).
4. Track heap watermark during sensor+export stress (continuous mode).
