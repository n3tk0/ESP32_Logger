Design audit · ESP32 Logger Web UI

Vanilla SPA · style.css 14.8 KB · index.html 90 KB · web.js 116 KB
12 ISSUES 5 HIGH 2 CRITICAL

Walked the SPA via the source — www/index.html, www/style.css, www/web.js. Firmware-side concerns are out of scope; the existing audit_webui.md covers those.

The redesign in the other tab is one possible response to the issues below — neutral slate panels, single accent, density toggle, real icons, tabular numerics on every value. It does not change a single API contract.

Critical · 2

A1 Critical
Single 90 KB index.html with 25+ pages inline
The entire SPA — every form, every settings subpage — is one HTML document. Editing one screen risks breaking all of them. The file is harder to diff than the firmware that serves it.
Split per route and inline at build time, or load partials over fetch. Keep the failsafe single-file fallback.
A2 Critical
Two duplicate id="db-error" elements on the dashboard
Lines 113 and 115 of index.html both declare id="db-error". Browsers ignore the second; whichever element web.js targets first wins, silently. Classic copy-paste bug.
Remove the duplicate. Add an HTML lint step to CI.

High · 5

B1 High
Emoji-as-icons everywhere
Navigation, buttons, badges and headings rely on system emoji (🌡️ 📊 ⚙️). They render differently on iOS, Android, Windows and the captive-portal browser. Some are plain symbols (▼) styled as chevrons — they cannot be tinted with CSS color.
Adopt a real icon set (Lucide / Material / Tabler SVG sprites). 2 KB inline sprite covers every glyph in the UI.
B2 High
Sidebar + bottom nav + header + page header — four chrome layers
Desktop renders the sidebar plus the page-header H1 plus a status pill row plus, on narrower widths, the mobile header. The actual content area gets ~55% of the viewport.
Pick one nav surface per breakpoint. Move device status into a single persistent topbar; demote the H1 since the active sidebar item already names the page.
B3 High
No information density — gigantic cards everywhere
--radius:12px, 1rem padding, 1.5rem page padding, generous margins between every card. A 24-inch monitor shows the same 4 KPIs as a phone. There is no compact mode for power users actually monitoring this thing.
Add a density toggle. Default to a tighter grid on desktop (12-col, 14 px gutters, 6 px radii).
B4 High
No tabular numerics anywhere — values jiggle when they update
Live values like 0.00 → 12.4 shift left/right because the default font has proportional digits. On a polling dashboard this looks broken.
Use font-variant-numeric: tabular-nums on every numeric. Pair with a real mono font for raw telemetry.
B5 High
Inline styles for everything custom
Hundreds of style="display:flex;gap:.5rem;..." attributes throughout index.html. New colors, new spacings, new font sizes appear on a per-element basis with no system.
Define utility classes or replace ad-hoc styling with composable components.

Medium · 5

C1 Medium
Color palette is one teal blob
Primary, accent and chart-FF are all variations of the same teal (#275673, #3182ce, #7eb0d5). State machine, badges and chart legend rely on color contrast that mostly does not exist.
One brand accent. State colors (ok/warn/err) should be perceptually distinct, not just hue-shifted teal.
C2 Medium
Native alert() for user feedback
Theme save, settings import, OTA upload — all surface errors via alert(). Acknowledged in the existing audit but still in the codebase.
Toast component already half-built in style.css; finish wiring it.
C3 Medium
State machine is invisible when stopped
Live page shows current state as a colored chip with a 2-line label. The pipeline diagram from README.md is more informative than the actual UI.
Render the state pipeline as a 5-step horizontal track with done/active/pending styling. The redesign shows it.
C4 Medium
Filters live in their own card, not on the chart
Date range, trigger filter, threshold and "exclude 0.00 L" each occupy a row in a dedicated card. By the time you scroll back to the chart, you have forgotten the filter values.
Inline the time-range as a segmented control on the chart card. Keep the date range for power users in a popover.
C5 Medium
Sensor cards have no signal of "is this fresh?"
.sensor-card shows the current value and a status badge but no timestamp, sparkline or staleness indicator. A frozen sensor looks identical to a healthy one.
Add "last update" timestamp + a 24-point sparkline + an err-state border for cards over their read_interval_ms threshold.

Low · 2

D1 Low
Light + dark themes share the same chart grid color
Chart.js uses default gridlines; in dark mode they are nearly invisible.
Theme tokens for chart axes, gridlines, tooltip background.
D2 Low
Form labels are plain text, hints are below the input
Every settings page has the same pattern: bold label, input, gray hint. The hint usually contains the most important info (default value, valid range). It is the smallest text on the page.
Hint above input, secondary; keep label as the primary anchor. Or use a placeholder with a real default.

Strengths · keep

  • Clean URL structure — hash-based SPA with predictable #dashboard, #settings_corelogic routes.
  • Failsafe recovery page — graceful degradation when /www/index.html is missing.
  • LTTB aggregation surfaced in the UI — power users can see and tune the visualisation pipeline.
  • Plugin-aware sensor architecture mirrored in the UI — Core Logic page maps 1:1 to platform_config.json.
  • Sensible separation of legacy / continuous / hybrid modes — UI does not hide complexity.
